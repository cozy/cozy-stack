<objective>
Run the litmus `props` suite (18 tests: begin, propfind_invalid, propfind_invalid2, propfind_d0, propinit,
propset, propget, propextended, propmove, propdeletes, propreplace, propnullns, prophighunicode, propremoveset,

**Critical uncertainty from research §7 pitfall I:** It is unknown empirically whether litmus treats a 501
Not Implemented response to PROPPATCH as a skip, a pass, or a failure. The first run reveals the answer.

**Three possible strategies depending on the first-run result:**

- **Strategy A (501 accepted):** PROPPATCH returns 501 with proper Allow header, litmus treats as expected
  behavior for a Class 1 server, all `prop*` tests that invoke PROPPATCH skip cleanly. Nothing to implement.
- **Strategy B (207 with 403 per-property):** PROPPATCH returns a valid `<D:multistatus>` body where every
  proposed property change is rejected with 403 Forbidden. Litmus accepts this as "server rejects dead properties".
  Requires a minimal PROPPATCH handler (~40 LoC) but no persistence layer.
- **Strategy C (real stub, last resort):** Implement a minimal in-memory dead-properties store. Expensive;
  fallback ONLY if A and B both fail the suite.

Also certain: `propfind_invalid`, `propfind_invalid2`, `propfind_d0`, `propwformed` — these exercise
PROPFIND (not PROPPATCH) and must return proper error responses (400 on malformed XML, no 500 crashes).
These are handled by plan 01-07's PROPFIND; if any fail, fix them here.

Purpose: Close the props portion of TEST-06 while keeping the PROPPATCH implementation as minimal as possible
(dead properties are a v2 requirement — ADV-V2-02).

Output:
- `03-07-INVENTORY.md` with first-run results + chosen strategy
- If Strategy B: `web/webdav/proppatch.go` + `proppatch_test.go` (new files, minimal 207-with-403 handler)
- If Strategy C: deeper implementation (unlikely — document heavily if it happens)
